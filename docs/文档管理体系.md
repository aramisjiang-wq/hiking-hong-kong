## 📋 文档分类说明

### 1. PRD (Product Requirements Document)
**产品需求文档目录**
- **用途**: 描述产品功能需求、用户故事、商业目标
- **包含内容**: 产品概述、功能模块、界面设计、用户流程、成功指标
- **更新频率**: 每次产品需求变更时
- **版本控制**: 语义化版本号 (v主版本.次版本.修订版本)

### 2. SPEC (Technical Specification)
**技术规范文档目录**
- **用途**: 详细技术实现方案、架构设计、API规范
- **包含内容**: 技术栈、数据库设计、API文档、安全策略、测试策略
- **更新频率**: 技术方案变更或重大架构调整时
- **版本控制**: 与PRD版本同步

### 3. MANUAL (操作手册目录)
**操作指南文档目录**
- **用途**: 提供详细的使用和操作指南
- **包含内容**:
  - `开发手册.md`: 开发环境搭建、开发流程、代码规范
  - `部署手册.md`: 部署流程、环境配置、运维指南
  - `用户手册.md`: 用户使用指南、功能说明

### 4. CHANGELOG (变更日志目录)
**项目变更记录目录**
- **用途**: 记录所有重要变更和版本发布信息
- **包含内容**:
  - `CHANGELOG.md`: 主要变更日志
  - `releases/`: 各版本详细变更记录

## 🔄 文档版本管理

### 版本命名规则
```
v主版本.次版本.修订版本-发布类型
```
- **主版本**: 重大功能更新或不兼容变更
- **次版本**: 新功能添加，向后兼容
- **修订版本**: Bug修复或小调整
- **发布类型**: alpha/beta/rc/stable（可选）

### 版本更新流程
1. **创建新版本目录** (如: `v2.1.0/`)
2. **更新版本号** (文档头部版本信息)
3. **记录变更** (CHANGELOG.md)
4. **审核确认** (负责人签字确认)
5. **发布通知** (通知相关人员)

## 🔄 文档更新流程

### 更新触发条件
1. **功能需求变更** → 更新PRD
2. **技术方案调整** → 更新SPEC
3. **操作流程变化** → 更新MANUAL
4. **新版本发布** → 更新CHANGELOG
5. **日常维护** → 按需更新

### 更新操作步骤
1. **检查现有文档**
   ```bash
   # 查看当前文档状态
   find docs/ -name "*.md" -exec echo "=== {} ===" \; -exec head -10 {} \;
   ```

2. **创建更新任务**
   - 使用todo_write工具创建任务
   - 设置合适的优先级和状态

3. **执行文档更新**
   - 按规范编写内容
   - 确保版本号更新
   - 添加变更记录

4. **质量检查**
   - 文档格式检查
   - 内容完整性验证
   - 版本一致性确认

5. **更新README**
   - 在README中添加文档信息
   - 更新文档状态和版本

## 📊 文档状态管理

### 文档状态定义
- **🟡 草稿** (Draft): 正在编写或修改
- **🟠 审核** (Review): 等待审核确认  
- **🟢 已发布** (Published): 审核通过，可正式使用
- **🔴 已废弃** (Deprecated): 被新版本替代，不再使用
